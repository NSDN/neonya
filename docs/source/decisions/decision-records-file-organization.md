# 2026-05-23 决策记录的文件组织形式

## 背景

在 session 中讨论产生的每个问题点，需要在文件系统中留痕——记载讨论脉络和最终决策，并确保后续 session 的 AI 在遇到歧义时能查阅这些记录。

## 讨论脉络

- **方案 A：一文件一次 session**
  - 优点：保留时间上下文，同轮讨论的多个决策可见关联。
  - 缺点：将来按主题检索时难以定位；修订某一决策时需在大篇中翻找。
- **方案 B：一文件一个主题**
  - 优点：每个决策独立可查，修订精准；文件命名直观。
  - 缺点：如果讨论中存在交叉引用，孤立文件会丢失部分讨论脉络。

- **存放位置**
  - `prd/`：产品需求文档，语义不符。
  - `archive/`：已封存的历史记录，不适合活跃决策。
  - 新建 `decisions/`：语义独立，便于在侧边栏单独导航。

- **AI session 间如何知晓**
  - 在 `AGENTS.md` 中追加 `Decision records` 段落，明确告知 AI 应先查阅该目录。

## 决策

采用 **一主题一文件**，统一放在 `docs/source/decisions/` 下。文件名使用 kebab-case，每篇遵循统一模板（背景 → 讨论脉络 → 决策 → 影响）。

## 影响

- 后续每次需要记录决策时，在 `docs/source/decisions/` 下按模板新建文件。
- VitePress 侧边栏新增「决策记录」导航入口。
- `AGENTS.md` 末尾追加 Decision records 说明，引导未来 session 查阅。
